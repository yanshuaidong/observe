## MVP 架构原则（先跑通再细化）
1. **Blob-first（后端不理解字段结构）**：`strategy` 表只保存策略的 JSON 整体内容，后端只校验“是否为合法 JSON”，不解析也不合并 `content` 内的字段结构。
2. **`daily_record` 不单独建表**：`daily_record` 作为 `strategy.content` 的一部分，由前端维护结构，并在每天更新时由前端提交“整份 `content`”覆盖保存。
3. **前端定制化展示**：因为不同策略 `content` 字段差异很大，前端先用“JSON 兜底编辑/显示”保证可用；后续再把每种策略替换成专属表单/展示组件。

## 技术栈与目录约定
- 后端：Python（FastAPI）
- 前端：Vue3 + Vite + Element Plus
- 数据库：MySQL（`JSON` 类型字段）
- 文档位置：需求与约束写在 `doc/架构设计.md` / `doc/初期需求文档.md`，字段约束写在 `database/策略1.md`

## 数据模型（MVP）
### 1) `strategy` 表（唯一表）
字段与约束（对齐 `database/策略1.md`）：
- `id`：主键（你不关心的数据库内部 id）
- `strategyid`：逻辑主键/唯一约束（建议由创建日期生成，例如 `20260330`）
- `content`：`JSON NOT NULL`，保存“该策略需要的所有信息”

`strategy.content` 最外层字段建议（**不强制**，但约定“可选”字段用于展示）：
- `name`：策略名称（如果存在，用于列表展示）
- `description`：策略说明（可选）
- `considerations`：策略应考虑哪些方面（可选）
- `execution`：策略执行/入场与处理思路（可选）
- `daily_record`：每日结构化记录（可选；结构由前端维护）

> 关键点：后端不依赖 `content` 内部结构是否包含上述键，只要它是合法 JSON，就可以保存并完整返回。

## 后端 API 契约（MVP：只做 strategy CRUD）
### 约定
- `strategy.content`：请求/响应均为 JSON 对象（客户端提交“整份内容”，服务端直接保存/返回）
- 错误约定（建议）：
  - `400`：参数错误（如 `strategyid` 为空）
  - `404`：策略不存在
  - `409`：`strategyid` 冲突（已存在）

### 1) 列表
`GET /api/strategies`
响应示例：
```json
{
  "items": [
    {
      "strategyid": "20260330",
      "content": { "name": "策略A" }
    }
  ]
}
```

### 2) 创建
`POST /api/strategies`
请求示例：
```json
{
  "strategyid": "20260330",
  "content": {
    "name": "策略A",
    "daily_record": {}
  }
}
```

### 3) 读取单个
`GET /api/strategies/{strategyid}`
响应示例：
```json
{
  "strategyid": "20260330",
  "content": { "name": "策略A", "daily_record": {} }
}
```

### 4) 更新（覆盖整份 content）
`PUT /api/strategies/{strategyid}`
请求与响应同“创建”，服务端以“整份替换”的方式保存 `content`。

### 5) 删除（可选）
`DELETE /api/strategies/{strategyid}`

## 前端交互职责（MVP）
1. `StrategiesPage`：
   - 调用 `GET /api/strategies`
   - 在表格中优先展示 `content.name`；若不存在则展示 `strategyid`
2. `StrategyEditor`（JSON 兜底）：
   - 输入 `strategyid`
   - 用 `textarea` 展示/编辑 `content` 的 JSON 文本
   - 保存时在前端校验 JSON（`JSON.parse`）
   - 保存成功后返回列表或保持当前页面

## 数据库建表 DDL（后续落地时会同步到代码/README）
（建议在第一次运行时执行）
```sql
CREATE TABLE IF NOT EXISTS strategy (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  strategyid VARCHAR(64) NOT NULL,
  content JSON NOT NULL,
  UNIQUE KEY uk_strategy_strategyid (strategyid)
);
```
